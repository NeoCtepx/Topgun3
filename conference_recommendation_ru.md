# Решение для конференции 16/32 участников с приватной видео-видимостью

## Краткий вывод

Ваш сценарий **невозможно реализовать в Zoom/Google Meet/Teams как стандартную «одну встречу»**:

- в этих продуктах нет штатной функции, где каждому участнику задается свой уникальный список видимых видеопотоков;
- при этом у вас нестандартное требование: **аудио от всех — всем**, а **видео — адресно и приватно**.

Это реализуется через **кастомную конференцию на WebRTC SFU** (селективная доставка потоков для каждого участника).

## Требования, как я их формализую

Для 16 участников:

- 4 «основных» участника (A1, A2, A3, A4);
- каждый из 4 должен видеть только 3 назначенных ему участников:
  - A1 видит группу G1 (3 человека),
  - A2 видит G2,
  - A3 видит G3,
  - A4 видит G4;
- группы G1..G4 должны быть взаимно эксклюзивными по видимости (никто, кроме «своего» основного участника, их не видит);
- аудио от всех 16 участников слышат все 16.

Масштаб до 32 участников:

- тот же принцип, только больше групп и/или больше «основных» участников;
- архитектура на SFU масштабируется до этого при правильной настройке.

## Что подойдет на практике

### 1) Лучший путь: кастомное приложение на готовом видеодвижке (SFU)

Подходящие платформы:

- **LiveKit (Cloud или self-hosted)**
- **Agora RTC**
- **100ms**
- **Dolby.io Communications API**
- **Twilio Programmable Video**

Почему именно они:

- позволяют подписывать каждого пользователя только на нужные видео-треки;
- позволяют оставить общий аудио-канал для всех;
- дают программный контроль логики (кто кого видит/слышит).

### 2) Что НЕ подойдет как готовая «кнопка»

- Zoom, Google Meet, Microsoft Teams, Discord (в стандартной форме) — не дают такой granular privacy-маршрутизации видео в одной комнате;
- «разбивка на 4 параллельные конференции» — вы это уже обозначили как нежелательный вариант.

## Рекомендуемый стек (практичный и быстрый)

- **SFU:** LiveKit Cloud (быстрый старт) или self-hosted LiveKit (если важен полный контроль).
- **Клиент:** веб-интерфейс (React/Vue/чистый JS), где для каждого пользователя задается список подписок на видео.
- **Сервер логики:** небольшой backend (Node.js/Python/Go), который хранит матрицу видимости и выдает права/токены.

## Логика маршрутизации

1. Все публикуют аудио + видео.
2. Сервер применяет правила:
   - аудио-треки: подписка `all -> all`;
   - видео-треки: подписка строго по матрице видимости.
3. Клиент отображает только разрешенные видео.
4. При необходимости оператор меняет матрицу «на лету».

## Сроки реализации (реалистично)

- MVP (16 участников, фиксированные группы): **3–7 рабочих дней**.
- Версия с панелью управления, логами, ролями и масштабом до 32: **2–4 недели**.

## Стоимость (ориентир)

- Если брать облачную платформу (LiveKit/Agora/100ms):
  - платите за минуты/участников/трафик;
  - старт для пилота обычно умеренный, но зависит от длительности сессий.
- Кастомная разработка:
  - MVP обычно дешевле, чем постоянные ручные костыли из нескольких Zoom-комнат;
  - при росте сценариев кастом быстро окупается.

## Прямой ответ на ваш вопрос

- **Да, задача решаема в рамках одной конференции.**
- **Нет, в «обычном Zoom» это не реализуется штатно.**
- Нужно решение класса **программируемой WebRTC-конференции (SFU)**.
